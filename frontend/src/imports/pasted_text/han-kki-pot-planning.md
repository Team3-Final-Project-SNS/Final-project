## 1. 프로젝트 개요

### 1. 프로젝트명

- **한끼팟**

### 2. 핵심 목적

- **대학 내의 사람들을 식사라는 매개로 연결한다.**

### 3. 타겟 사용자

- 첫 진입 대학 관계자
- 학식, 점심, 저녁 메이트를 찾는 대학생
- 초기 핵심 타겟 시점
    - 신학기 (3월, 9월)
        - 새 인간관계 형성 니즈 최고조 → 신입생 집중 유입
    - 복학생
        - 학교 복귀 후 아는 사람 적음 → 자연스러운 유입 시나리오
    
    → 온보딩 문구 / 마케팅을 시점 별로 다르게 구성
    

### 4. 플랫폼

- **Web 기반** (반응형 디자인)

### 5. 차별 포인트

- **책임비 포인트 예치 시스템** — 게시글 작성·매칭 시 포인트를 예치하여 노쇼를 구조적으로 억제
- **QR 기반 만남 인증** — 만남 사실을 객관적으로 검증
- **학교 이메일 기반 폐쇄형 커뮤니티** — `.ac.kr` 인증으로 신뢰 확보

## 2. 핵심 사용 시나리오

```java
[A: 24학번, 1시 반에 수업 끝남]
   ↓
게시글 작성: "밥 먹을 사람? 1시 반에 수업 끝남"
+ 책임비 500P 설정 (등록자 500P 예치)
   ↓
[B: 22학번 선배가 신청]
   ↓
B가 게시글에 신청 → 동일 500P 예치 → 선착순 매칭 확정
   ↓
1:1 채팅방 자동 생성
   ↓
"수업 어디서 끝나? 어디서 만나자"
"뭐 먹을래요?"
"22학번인 내가 추천해줄게, 여기 가봤어?"
   ↓
오프라인 만남
   ↓
QR 인증 1차/2차 (장소인증 / 만남인증 / 등록자 표시 → 신청자 스캔)
   ↓
만남 완료 → 예치 포인트 양쪽 반환
   ↓
식대 N빵 (오프라인 정산)
```

## 3. MVP 핵심 기능

### 1. 학교 이메일 가입

- **학교 이메일(.ac.kr) 기반 OTP 인증**
- 한 학교 이메일당 한 계정
- 가입 시 이름·닉네임·학과·학번·생년·성별 입력
- 가입 흐름
    1. 이메일 입력 → OTP 발송
    2. OTP 검증 + 회원 정보 입력
    3. 약관 동의 → 가입 완료

---

### 2. 게시글

### 1. 게시글 작성

- 입력 항목
    - 만남 희망 시간 (필수)
    - 만남 장소 선택(필수) - 정문, 후문, 기숙사, 동아리관 → 추후 장소 선택 기능으로 고도화
    - 희망 식당 (선택)
    - 한마디 (수업 끝나는 시간 등 하고싶은 말)
    - **책임비 포인트 설정** (최소 200P, 본인 보유 한도 내 자유)
- 작성 즉시 책임비 포인트 예치 (잔액 차감, point_transaction에 DEPOSIT 기록)
- 잔액 부족 시 작성 불가

---

### 2. 게시글 목록 조회 / 상위 노출

- 같은 학교 게시글 노출 (작성자의 school_id 기준)
- **책임비 포인트가 높은 게시글이 상위 노출**
- 게시글 상태값 추가 → 상태값으로 필터링

---

### 3. 게시글 수정 / 삭제

- 본인 게시글만 가능
- **`OPEN` 상태일 때만** 수정·삭제 가능 ( 매칭 전에만 수정 삭제 가능)
- 삭제 시 예치 포인트 반환

---

### 3. 신청 / 매칭 (1:1 선착순)

- 신청자가 상세페이지에 들어가서 "신청하기" 클릭
- **등록자와 동일한 예치 포인트 차감**
- **선착순 매칭 - 고도화에서 동시성 적용 (Redis 분산락, DB 비관락 등)**
- 본인 게시글 신청 불가

---

### 4. 채팅 (WebSocket / STOMP)

- 매칭 확정 시 자동 1:1 채팅방 생성
- 실시간 채팅 대화로 메뉴·식당 등 세부 사항 확정
- 최종 만남 시간 +2h 후 채팅방 비활성화 (조회만 가능) → 일방적인 연락 방지

---

### 5. 오프라인 인증

- 장소 인증 (GPS)
    - 만남 장소에서 GPS 반경 50m 기반 위치 인증
    - 양측 모두 50m 이내 진입 확인 시 장소 인증 완료
    - 만남 시간 15분전 ~ 1시간까지 활성화
    - author_place_verified_at / applicant_place_verified_at 로 변경
    - 지도 UI: 약속 장소 반경 내 위치와 상대방 위치만 표시 (파란 점)
- 만남 인증 (QR)
    - 장소 인증 완료 후 QR 인증 단계 진입
    - 매칭 시 서버가 등록자에게 만남 인증 QR 토큰 생성 (TTL: 만남 시간 + 0.5h)
    - 등록자가 QR 표시 → 신청자가 스캔
    - 만남 인증 완료 시 정상 만남.
- 인증 시각 분리 저장(`author_displayed_at`, `applicant_scanned_at`) → **노쇼 정택 참조**

### 6. 포인트 시스템

| 이벤트 | 변동 |
| --- | --- |
| 가입 시 | +10,000P |
| 게시글 작성 시 | 설정한 책임비 포인트 예치 |
| 신청자 매칭 시 | 등록자와 동일 포인트 예치 |
| 만남 정상 완료 | 예치 포인트 전액 반환 |
| 약속시간 이전 취소 | 취소한 사용자의 예치 포인트 50%만 반환 |
| 노쇼 | 노쇼한 사용자 예치 포인트 차감 |

| 타입 | 부호 | 설명 |
| --- | --- | --- |
| `SIGNUP_BONUS` | + | 가입 보너스 (10,000P) |
| `DEPOSIT` | - | 게시글 작성·매칭 시 예치 |
| `REFUND` | + | 정상 완료 또는 사전 취소 반환 |
|  | + | 매칭 취소 시 50% 반환 |
| `FORFEITURE` | - | 노쇼 시 환수 |

### 정합성 보장

- `user.point`는 **캐시 컬럼**이며, 실제 정합성은 `point_transaction` 누계로 검증한다.
- 모든 거래 직후 `balance_after`를 함께 기록하여 추적 가능하도록 한다.
- 동시 거래에 대비해 `user.point` 갱신은 비관락 또는 낙관락 사용 (Redis 분산락 보조 가능).

### 잔액 부족 처리

- 게시글 작성·신청 시점에 잔액 검증
- 잔액 부족 시 `POINT_001` 에러 반환
- 자연 회복 또는 충전 시스템은 Phase 2 단계

---

### 7. 회원가입 시 약관 동의

- 우리 서비스의 목표는 순수하게 밥약을 잡는 용도입니다.
- **만남에서 발생한 식대는 만나서 N빵이 기본 원칙입니다.**
- 부적절한 게시물은 신고의 대상이 되며 관리자가 임의로 삭제 할 수 있습니다. → 신고 포상금 50P
- 용도 이외 사용 시 벌어지는 문제에 대해 당사는 법적 책임지지 않습니다.

## 백엔드 기능 별 MVP 최적화

### 1. 동시성 제어

- **기술**: Redis 분산락, DB 비관락
- **적용 위치**: 선착순 매칭, 포인트 차감
- **내용**
    - MVP: `post_id` UNIQUE 제약으로 1차 방어
    - 고도화: Redis 분산락(`lock:post:{postId}`)으로 동시 요청 직렬화
    - 트랜잭션 내 게시글 상태 변경 + 매칭 생성 + 포인트 차감 + 채팅방 생성 원자적 처리
    - 동시 요청 테스트 코드 작성 및 결과 문서화

---

### 2. 성능 최적화 (부하 테스트 + 성능 병목 개선 + 캐싱)

- **기술**: Redis Cache, QueryDSL, Index 튜닝, K6
- **적용 위치**: 게시글 목록 조회, 인기 게시글 상위 노출
- **내용**
    - Redis ZSet으로 책임비 포인트 기반 게시글 상위 노출
    - QueryDSL 동적 쿼리 + 페이징 최적화
    - EXPLAIN 기반 실행 계획 분석 → 인덱스 추가
    - K6 부하 테스트 → Latency(P95/P99), Throughput(RPS), Error Rate 기록
    - 개선 전후 수치 비교 문서화

---

### 3. 로그·모니터링 (Phase 1 고도화 대상)

- **기술**: Logback / SLF4J, Prometheus + Grafana 또는 CloudWatch
- **적용 위치**: 전체 API, 포인트 거래, 노쇼 판정, GPS/QR 인증
- **내용**
    - 로그 수집 환경 구성 (구조화 로그, 로그 레벨 정책)
    - 모니터링 지표 선정 및 수집 근거 문서화
    - **한끼팟 특화 수집 지표**
        
        
        | 지표 | 인사이트 |
        | --- | --- |
        | 게시글 작성 → 매칭 전환율 | 게시판 활성화 여부 판단 |
        | 노쇼 발생 비율 | 포인트 정책 조정 근거 |
        | GPS 인증 실패율 | 반경 50m 기준 적정성 검토 |
        | 포인트 잔액 부족 에러 빈도 | 포인트 충전 기능 도입 시점 판단 |
        | 매칭 후 채팅 활성화율 | 서비스 실사용 품질 측정 |
    - 장애/이상 상황 감지
        - 포인트 이중 차감 감지
        - QR 토큰 만료 전 인증 실패 급증 알림
        - Redis 연결 장애 감지

## 백엔드 기능 별 고도화

### 1. 실시간 메시징

- **기술**: WebSocket / STOMP + Redis Pub/Sub
- **적용 위치**: 1:1 채팅
- **내용**
    - 메시지 브로커로 Redis Pub/Sub 적용 → 멀티 인스턴스 환경 메시지 분배
    - 인증된 사용자만 채팅방 접근 가능한 구조 보장
    - 채팅 메시지 `chat_message` 테이블 영구 저장

---

### 2. 트래픽 제어

- **기술**: Rate Limiting (Redis 기반 또는 Spring 레벨)
- **적용 위치**: 신청하기 API, OTP 발송 API
- **내용**
    - 신청 API 과도한 요청 제어
    - OTP 발송 어뷰징 방지
    - 임계값 설정 근거 문서화
    - 초과 요청 시 `429 Too Many Requests` 응답 처리

---

### 3. 배포 고도화

- **기술**: AWS EC2 멀티 인스턴스, GitHub Actions CI/CD
- **내용**
    - 단일 인스턴스 → 다중 인스턴스 전환 시 Redis Pub/Sub 필요성 연결
    - 무중단 배포 전략 (Blue-Green 또는 Rolling)
    - 배포 전후 가용성 비교 문서화

## AI 기능 별 MVP

### 1. LLM 구조화 출력 + 프롬프트 엔지니어링

- **적용 후보**: 게시글 한마디 자동 생성 보조 / 노쇼 이의제기 사유 분석
- **내용**
    - LLM 응답을 DTO/JSON Schema로 파싱 → DB 저장
    - `max_tokens`, `temperature` 명시 설정 및 근거 문서화
    - 도메인별 프롬프트 템플릿 최소 2개 분기 (예: 신입생용 / 복학생용 온보딩 메시지)
    - 프롬프트 코드 하드코딩 X → 서버 재배포 없이 수정 가능한 구조
    - 프롬프트 개선 이력 최소 3회 기록

---

### 2. Tool Calling 기반 AI-백엔드 API 연동

- **적용 후보**: AI 밥약 어시스턴트
- **시나리오 예시**
    
    ```
    사용자: "오늘 12시에 정문 근처에서 밥 먹을 사람 있어?"→ LLM이 searchPosts(time="12:00", place="정문") 호출→ LLM이 checkUserPoint(userId) 호출 → 포인트 충분한지 확인→ "현재 12시 정문 게시글 2개가 있어요. 포인트도 충분하니 바로 신청할 수 있어요!"[장애 상황] searchPosts API 타임아웃→ "지금 게시글 조회가 어려워요. 잠시 후 다시 시도해주세요."
    ```
    
- **Tool 후보**: `searchPosts` / `checkUserPoint` / `getMatchStatus`

---

### 3. AI 기능 장애 격리 + 비용 인식 설계

- **내용**
    - LLM API 장애 시 핵심 기능(매칭, 포인트, 채팅)에 영향 없어야 함
    - Fallback 최소 2가지
        - 타임아웃(5초 초과) → 캐시된 응답 반환 또는 안내 메시지
        - Rate Limit(429) → Circuit Breaker 오픈, 기본 응답 반환
    - 환각 방어 1가지 이상 (존재하지 않는 게시글 정보 생성 시 차단)
    - 비용 로깅 항목: `requestId`, `promptTokens`, `completionTokens`, `latencyMs`, `model`, `status`, `errorType`
    - 토큰 사용량·응답 지연·에러율 대시보드 구성

## 5. MVP 정책

### 1. 가입

- 학교 이메일(.ac.kr)
- 한 이메일당 한 계정

### 2. 게시글

- 최소 예치 포인트 200P
- 책임비 포인트 자율 설정 (본인 보유 한도 내)
- 매칭 후 만남 희망 시간 이전까지 15분 1회 연장 가능

### 3. 매칭

- 1:1 선착순 매칭
- 신청 시(＝매칭 시) 등록자 예치 포인트와 동일한 포인트 차감

### 4. 포인트

- 가입 시 10,000P
- 게시글 작성·신청 시 예치
- 예치 포인트는 100P단위로 증감
- 정상 완료 시 예치 포인트 100% 반환
- 매칭 취소 시 예치 포인트 50% 반환
- 노쇼 시 예치 포인트 전체 차감
- 잔액 부족 시 게시글 작성·신청 불가(포인트 충전 기능 구현 예정)

### 5. 식대

- N빵 기본 원칙
- 회원가입 약관 동의로 명시

### 6. 안전 (약관 명시)

- 시스템 강제 정산 X (오프라인 협의)
- 만남 장소는 공공장소 권장
- 회원가입 시 약관 동의 진행

### 7. 노쇼 정책

> 
> 
> 
> **도덕적 판단이 아닌 시스템 규칙 기반 자동 판정**
> 
> 분쟁이 발생하지 않도록 **규칙 자체가 곧 판정 기준**이 된다. 사용자는 규칙을 알고 따르기만 하면 안전하다. 규칙을 어긴 쪽이 노쇼로 처리된다.
> 

```java
0. 해피 패스
		A가 장소 QR을 인증하고 10분 이내에 B도 QR을 인증. 
		+ 상호 만남 QR을 인증하면 만남 성사 완료 -> 예치포인트 반환

1. 약속시간 이전에 사전 취소하는 경우 -> 예치 포인트의 50%만 반환
2. 약속시간 이전에 채팅으로 늦음 공지 -> 약속시간 이전에만 15분 1회 연장 가능
3. 사전 늦음 공지를 하지 않고 한명만 인증한 경우 -> 노쇼한 사용자 예치포인트 반환 없음
	3-1. A가 장소 QR을 인증하고 B가 인증하기 전에 그 반경을 벗어나면 노쇼로 판정
	3-2. A가 장소 QR을 인증하고 10분이 지나면 B가 노쇼로 판정
	3-3. A가 장소 QR을 인증하고 10분 이내에 B도 QR을 인증. 
					상호 QR을 인증하기 전에 먼저 반경을 벗어난 사용자가 노쇼로 판정
4. 최종 약속시간 이후 30분간 아무도 인증하지 않음 -> 둘다 노쇼

억쇼 방지
- 억울한 노쇼 방지하기 위해, 노쇼 예정 상태임을 안내.
  24시간 안에 이의제기가 없을 경우 노쇼로 판정을 한다.
	  ex. "'닉네임'님은 현재 'match_id'만남에서 노쇼 예정 상태입니다. 동의하십니까?
	  해당 사안에 대해 24h안에 응답이 없을 경우 예치 포인트가 차감됨을 알려드립니다"
```

### 장소 인증 단계 노쇼 판정 (GPS 기반)

| 등록자 GPS 진입 | 신청자 GPS 진입 | 판정 |
| --- | --- | --- |
| ✅ | ✅ | 장소 인증 완료 → 만남 QR 단계 진입 |
| ✅ | ❌ | `APPLICANT_NO_SHOW` (장소 미도달) |
| ❌ | ✅ | `AUTHOR_NO_SHOW` (장소 미도달) |
| ❌ | ❌ | `BOTH_NO_SHOW` |
- GPS 반경: 약속 장소 기준 **50m**
- 만남 시간 15분 전 ~ 1시간 이후 까지 장소 인증 GPS 활성화
- 만남인증이 완료되면 GPS 비활성화

---

### 만남 QR 단계 노쇼 판정

| 경우의 수 | 판정 |
| --- | --- |
| 스캔 완료 | `COMPLETED` |
| 30분 후 신청자가 없음 | `APPLICANT_NO_SHOW` |
| 30분 후 등록자가 없음 | `AUTHOR_NO_SHOW` |
- QR 토큰의 유효 기간(`expires_at`)은 **장소인증이 된 시간 부터 30분**이며, 이 시간이 경과하면 `EXPIRED` 처리된다.

### 억울한 노쇼 방지 (이의 제기 절차) → 장소인증 완료자만 해당

- 노쇼 예정으로 먼저 분류 (`_NO_SHOW` 상태)
- **24시간 이내 이의 제기가 없으면** 노쇼 확정 및 예치금 환수
- 이의 제기 시 작성한 사유를 바탕으로 관리자 검토 후 최종 판정 (채팅 기록 등 보존된 자료 활용)
- 노쇼 당한 사람끼리 같은 반경안에 있으면 매칭 추천해주는 기능 도입 예정

## 9. 주요 설계 포인트

**선착순 매칭 동시성**

- 신청 요청에 대해 `match` row 생성 시 `post_id`에 UNIQUE 제약으로 1차 방어
- Redis 분산락(`lock:post:{postId}`)으로 동시 요청 직렬화
- 트랜잭션 내에서 게시글 상태 변경 + 매칭 생성 + 포인트 차감 + 채팅방 생성을 원자적으로 처리

**실시간 채팅**

- WebSocket / STOMP 기반
- 메시지 분배에 Redis Pub/Sub (멀티 인스턴스 대비)
- 채팅 메시지는 `chat_message` 테이블에 영구 저장

**스케줄러**

- 만남 시간 +30분 시점에 자동 노쇼 판정
- Redis Key TTL + Keyspace Notification 또는 Spring `@Scheduled` 활용

**포인트 정합성**

- 모든 포인트 변동은 트랜잭션 내에서 `point_transaction` 기록과 함께 발생
- 분쟁 시 `point_transaction` 누계로 `user.point` 검증
